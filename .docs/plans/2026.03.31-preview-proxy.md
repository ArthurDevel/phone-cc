# Preview Proxy: Token-Gated Reverse Proxy for Agent Dev Servers

## High-Level Goal

When a Claude agent starts an HTTP server (e.g. `localhost:3002`), the user cannot reach it because only port 3000 is exposed. This feature adds a transparent reverse proxy through the Next.js app: agent output containing `localhost:XXXX` is automatically rewritten to a token-gated preview URL (`/preview/{token}/...`), and user messages containing preview URLs are rewritten back to `localhost:XXXX` before reaching the agent. The agent never knows about the proxy.

## Important Technical Notes

### URL Pattern Matching

The rewrite regex must handle these variations without false positives:
- `http://localhost:3002` (full URL)
- `https://localhost:3002` (rare but possible)
- `localhost:3002` (no protocol)
- `http://127.0.0.1:3002`
- `http://0.0.0.0:3002`

It must NOT match:
- `localhost` without a port (that's port 80, which is the Next.js app itself)
- Ports below 1000 (4+ digit requirement in regex naturally excludes these; the proxy route additionally rejects ports below 1024)

Pattern: `(?:https?://)?(?:localhost|127\.0\.0\.1|0\.0\.0\.0):(\d{4,5})(/[^\s)\"'>]*)?`

### Proxy Route: Asset Path Rewriting

When a dev server returns HTML, relative asset paths (e.g. `/_next/static/...`, `/favicon.ico`) will break because the browser resolves them against `/preview/{token}/`. The proxy route must handle this by:
- Forwarding the request path after the token prefix to `localhost:PORT`
- This naturally handles asset requests: `/preview/{token}/_next/static/foo.js` -> `localhost:PORT/_next/static/foo.js`

The HTML itself may contain absolute paths (`src="/app.js"`). We do NOT rewrite HTML content -- this would be fragile and slow. Instead, the proxy relies on the catch-all route handling all sub-paths. If a dev server uses absolute paths like `/app.js`, the browser will request `/app.js` (not `/preview/{token}/app.js`), which will 404. This is an accepted limitation for v1. A `<base>` tag injection could solve this later.

### Known Limitations (v1)

- **No WebSocket proxying.** Dev server HMR (hot module reload) will not work through the proxy. The user must manually refresh to see changes.
- **Absolute paths in HTML.** Dev servers that use absolute paths (e.g. `src="/app.js"` instead of `src="./app.js"`) will break because the browser resolves them against `/` instead of `/preview/{token}/`.

### Host Header

The proxy must set the `Host` header to `localhost:{port}` when forwarding, so dev servers that check the Host header don't reject the request.

### SSRF Prevention

The proxy must reject requests to port 3000 (the Next.js app itself) to prevent request loops. Only ports 1024-65535 are allowed, excluding the app's own port.

### Hop-by-Hop Headers

The proxy must strip hop-by-hop headers (`Transfer-Encoding`, `Connection`, `Keep-Alive`, `Upgrade`) from the proxied response before forwarding to the client.

### Auth Protection

The `/preview/...` routes are already covered by the existing auth proxy in `src/proxy.ts` (its matcher catches all routes except explicit exclusions). Preview routes must NOT be added to the auth exclusion list -- the auth cookie protects them in addition to the token.

### Rewriting Strategy: Full Messages Only (No SSE Delta Rewriting)

Text deltas from SSE arrive as small fragments (e.g. `"local"`, `"host:"`, `"3002"`). The URL may be split across multiple deltas. Rewriting deltas in real-time is fundamentally broken: you cannot un-emit characters already sent to the client when a URL match spans multiple deltas.

Instead, we rewrite at two points only:
1. **On `message_end`** (in session-manager): rewrite the full `currentAssistantMsg.content` before emitting the `message_end` event. This means the stored message has rewritten URLs.
2. **History endpoint**: rewrite each assistant message's content before returning. This handles reconnection and page refresh.

The user will see raw `localhost:XXXX` URLs while the agent is streaming, then the correct preview URL once the message completes. On page refresh, all URLs in history are rewritten. This is acceptable for v1.

Note: `rewriteAgentOutput` in the history endpoint creates tokens as a side-effect for any new ports it encounters. This means tokens are not stable across server restarts -- a reconnected session will get new tokens with different values. Functionally this is fine since the proxy still works.

## Per-File Methods and Types

### preview-manager.ts (NEW)

File description: Manages preview tokens that map random IDs to localhost ports, enabling token-gated reverse proxying of agent-started dev servers. Handles URL rewriting in both directions (agent->user and user->agent).

Follow `code_formatting.md` conventions: JSDoc on all functions, code section headers, file-level description.

#### Types

```
PreviewToken {
  token: string
  port: number
  sessionId: string
  createdAt: Date
}
```

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `rewriteAgentOutput` | `sessionId: string, text: string` | `string` | Finds localhost URLs in text, creates tokens for new session+port pairs (reuses existing), replaces with preview URLs |
| `rewriteUserInput` | `text: string` | `string` | Finds preview URLs in text, replaces with the original localhost URL. Unknown tokens are left as-is |
| `lookupToken` | `token: string` | `PreviewToken \| undefined` | Returns the token entry for a given token string. Used by the proxy route |
| `cleanupSession` | `sessionId: string` | `void` | Removes all tokens belonging to a session |

### route.ts (preview/[token]/[[...path]]) (NEW)

File description: Catch-all reverse proxy that forwards HTTP requests to localhost ports based on preview tokens.

Use `RouteContext<"/preview/[token]/[[...path]]">` for the route context type.

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `GET` | `request: Request, ctx: RouteContext` | `Response` | Looks up token, proxies GET request to localhost:port/path, streams response back. Returns 404 for invalid tokens |
| `POST` | `request: Request, ctx: RouteContext` | `Response` | Same as GET but for POST requests, forwards request body |
| `PUT` | `request: Request, ctx: RouteContext` | `Response` | Same as GET but for PUT requests |
| `PATCH` | `request: Request, ctx: RouteContext` | `Response` | Same as GET but for PATCH requests |
| `DELETE` | `request: Request, ctx: RouteContext` | `Response` | Same as GET but for DELETE requests |

All methods use a shared `proxyRequest` helper.

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `proxyRequest` | `request: Request, token: string, pathSegments: string[] \| undefined` | `Response` | Core proxy logic: validates token, joins pathSegments with "/" (defaults to "" when undefined), builds target URL, forwards request with headers (sets Host to localhost:port, strips hop-by-hop headers), streams response back. Rejects port 3000 and ports <1024. Returns 404 for invalid tokens |

### message/route.ts (MOD)

Change: call `rewriteUserInput(text)` on the user's message text before passing to `sendMessage()`.

### session-manager.ts (MOD)

Changes:
- In `sendMessage()`, before emitting `message_end`: rewrite `currentAssistantMsg.content` using `rewriteAgentOutput(sessionId, content)`.
- In `closeSession()`: call `cleanupSession(id)` to remove all preview tokens for the session.

### history/route.ts (MOD)

Change: before returning messages, rewrite each assistant message's `content` field using `rewriteAgentOutput(sessionId, content)`.

- Run tests, ensure they pass

### Phase 3: Implement proxy route handler
- Create `src/app/preview/[token]/[[...path]]/route.ts`
- Implement `proxyRequest` helper and export GET, POST, PUT, PATCH, DELETE handlers
- Write integration tests in `src/__tests__/preview-proxy.test.ts` (spin up a simple http server in the test)
- Run tests, ensure they pass

### Phase 4: Wire up rewriting
- In `session-manager.ts` `sendMessage()`: rewrite `currentAssistantMsg.content` before emitting `message_end` (guard with `if (currentAssistantMsg)` -- it can be null if the turn had only tool use)
- In `session-manager.ts` `closeSession()`: call `cleanupSession(id)`
- In `message/route.ts`: call `rewriteUserInput(text)` before `sendMessage()`
- In `history/route.ts`: rewrite assistant message content before returning

### Phase 5: Build and verify
- Run `pnpm build` -- must pass with zero errors
- Run `pnpm test` -- all tests must pass
- Manual verification: start dev server, create a session, have the agent start a server, verify the URL in chat is rewritten and the proxy serves the content

## Success Criteria

1. Agent output containing `localhost:XXXX` is automatically replaced with a preview URL in the chat (after message completes)
2. User messages containing preview URLs are transparently converted back to `localhost:XXXX` before reaching the agent
3. Preview URLs proxy HTTP requests to the correct localhost port
4. Invalid/expired tokens return 404
5. Tokens are cleaned up when sessions are closed
6. Same port in the same session reuses the same token
7. Different sessions get different tokens even for the same port
8. Port 3000 is rejected by the proxy (SSRF prevention)
9. `pnpm build` passes with zero errors
10. All tests pass
