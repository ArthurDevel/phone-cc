# Feature 8: Session Persistence & Reconnection -- Implementation Plan

## Overview

Make sessions survive server restarts and browser reloads. The backend already persists session metadata (`session.json`) and SDK session IDs (`.sdk-session-id`) to disk. The `reconnectSession()` function already reads these and creates agent entries with `resume`. What's needed: auto-reconnect on session switch, localStorage for last active session, and robustness.

---

## Step 1: Auto-Reconnect on Session Switch

**Modify:** `src/contexts/session-context.tsx`

### Changes
- In `switchSession()`, check if the session's status is `disconnected`
- If disconnected, call `POST /api/sessions/[id]/reconnect` before switching
- Update the session's status in state to `active` after successful reconnect
- Store `activeSessionId` in `localStorage` key `phonecc:lastActiveSessionId`
- On initial load, read from localStorage and if the session exists, switch to it

---

## Step 2: Verify Backend Robustness

**Review:** `src/lib/session-manager.ts`

The `listSessions()` function already:
- Reads directory entries from `~/.phonecc/sessions/`
- Skips entries where `session.json` is missing/corrupt (returns null)
- Returns sessions with status based on whether they're in the agents map

The `reconnectSession()` function already:
- Reads session metadata and SDK session ID from disk
- Creates an agent entry with the stored SDK session ID for resume

No changes needed to session-manager -- it already handles this correctly.
