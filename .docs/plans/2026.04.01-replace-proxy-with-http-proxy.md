# Fix Preview Proxy: HTML Path Rewriting + Error Handling

## High-Level Goal

The current preview proxy (`route.ts`) uses `fetch()` to forward requests to agent dev servers. The forwarding itself works, but two issues make it unusable:

1. **Broken assets**: Upstream HTML contains absolute paths (`/_next/static/...`, `/favicon.ico`) that the browser resolves against PhoneCC's origin instead of going through the proxy. Result: "This page isn't working."
2. **No error propagation**: When `fetch()` throws (upstream down), there's no try/catch, so Next.js returns a generic 500.

Fix both by keeping the existing `fetch()` proxy and adding:
- HTML path rewriting: rewrite absolute paths (`src="/foo"`, `href="/foo"`) in HTML responses to route through `/preview/{token}/foo`
- Try/catch with 502 response on upstream connection failure
- Query string forwarding (currently dropped)
- `Location` header rewriting on redirects

No new dependencies needed.

## Important Technical Notes

### HTML path rewriting (not `<base>` tag)

**Why not `<base>`**: The `<base href>` tag only affects relative URLs (`./foo`, `foo`). It does NOT affect absolute paths starting with `/`. Since Next.js generates absolute paths like `/_next/static/...`, a `<base>` tag would not fix the core problem.

**Approach**: For HTML responses, read the body as text and rewrite absolute paths in `src` and `href` attributes to route through the proxy:
- `src="/_next/static/foo.js"` becomes `src="/preview/{token}/_next/static/foo.js"`
- `href="/styles.css"` becomes `href="/preview/{token}/styles.css"`

Use a regex that targets `src="/"` and `href="/"` patterns in HTML attributes. This is intentionally narrow to avoid false positives — we only rewrite paths that start with `/` inside `src` and `href` attributes.

Pattern: `((?:src|href|action)=["'])\/(?!\/)`  replace with  `$1/preview/{token}/`

This rewrites `src="/foo"` to `src="/preview/{token}/foo"` but does NOT touch `src="//cdn.example.com"` (protocol-relative URLs) or `src="https://..."` (full URLs).

After modifying the HTML body, delete the `Content-Length` header (the body length changed) and delete `Content-Encoding` (the response was decompressed by `fetch()` when reading as text).

### Inline script/JSON references

HTML may contain absolute paths in inline `<script>` blocks (e.g., `fetch("/api/data")`). The regex approach intentionally does NOT rewrite these — it is too risky to modify JavaScript source. This means API calls from inline scripts will bypass the proxy. This is an accepted limitation for v1. The catch-all route structure (`[[...path]]`) does handle these requests if the browser resolves them under `/preview/{token}/`, but inline `fetch("/api/data")` will go to PhoneCC's `/api/data` endpoint instead. A future improvement could use a Service Worker.

### Query string forwarding

The current code builds `targetUrl` from `pathSegments` only, dropping query strings. Fix: extract the search portion from `request.url` and append it to `targetUrl`.

### Location header rewriting on redirects

When the upstream returns a 3xx with a `Location` header containing an absolute path (e.g., `/dashboard`), the browser would follow it to PhoneCC's `/dashboard` — bypassing the proxy. Rewrite `Location` headers:
- Absolute paths (`/foo`) get `/preview/{token}` prepended
- Full localhost URLs (`http://localhost:{port}/foo`) get replaced with `/preview/{token}/foo`

### Content-Length and Content-Encoding after HTML modification

After reading the response as text (which decompresses it) and modifying it, the original `Content-Length` and `Content-Encoding` headers are stale. Delete both from the response headers before returning.

### APP_PORT hardcoding

The current code hardcodes `APP_PORT = 3000`. Since Next.js may start on a different port, use `parseInt(process.env.PORT || "3000", 10)`.

### WebSocket limitation

WebSocket connections (used by Next.js HMR) will not work through this proxy. Live-reload will be broken for previewed apps. Users must manually refresh.

## Per-File Methods and Types

### route.ts (preview/[token]/[[...path]]) (MOD)

File description: Catch-all reverse proxy that forwards HTTP requests to localhost ports based on preview tokens. Rewrites absolute paths in HTML responses and redirect headers so assets and navigation stay within the proxy.

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `proxyRequest` | `request: Request, token: string, pathSegments: string[] \| undefined` | `Promise<Response>` | Validates token, checks port safety, builds target URL (including query string), fetches from upstream with error handling. For HTML responses, rewrites absolute paths. For redirects, rewrites `Location` header. Returns 404 for invalid tokens, 502 for upstream errors |
| `rewriteHtmlPaths` | `html: string, token: string` | `string` | Rewrites absolute paths in `src`, `href`, and `action` attributes to route through `/preview/{token}/`. Skips protocol-relative and full URLs |
| `rewriteLocationHeader` | `location: string, token: string, port: number` | `string` | Rewrites a Location header value: absolute paths get `/preview/{token}` prepended, `localhost:{port}` URLs get replaced with the proxy path |

### preview-manager.ts (MOD)

Changes: remove the debug `console.log` statements from `rewriteAgentOutput`.

## Phases

### Phase 1: Fix proxyRequest error handling + query strings
- Wrap `fetch()` in try/catch, return 502 with error message on failure
- Extract query string from `request.url` and append to `targetUrl`
- Replace hardcoded `APP_PORT = 3000` with `parseInt(process.env.PORT || "3000", 10)`

### Phase 2: Add HTML path rewriting
- After `fetch()`, check if response `Content-Type` includes `text/html`
- If HTML, read body as text, rewrite absolute paths in `src`/`href`/`action` attributes
- Delete `Content-Length` and `Content-Encoding` headers from modified response
- Return modified response with same status and updated headers

### Phase 3: Add Location header rewriting
- For 3xx responses, check `Location` header
- If it starts with `/`, prepend `/preview/{token}`
- If it contains `localhost:{port}`, replace with `/preview/{token}`

### Phase 4: Clean up
- Remove debug console.logs from `preview-manager.ts`

### Phase 5: Update tests
- Add tests for: HTML path rewriting, query strings, 502 on connection refused, redirect rewriting (absolute path + localhost URL), non-HTML passthrough
- Keep existing tests (404, GET/POST forwarding, SSRF, hop-by-hop)
- Run all tests, ensure pass

### Phase 6: Build and verify
- `pnpm build` must pass
- `pnpm test` must pass
- Manual: navigate to a preview URL, verify page renders with assets loading correctly

## Success Criteria

1. Navigating to `/preview/{token}` renders the upstream page correctly, including JS/CSS/image assets
2. Upstream errors (like Supabase missing-env) display properly in the browser
3. API requests from the upstream app work through the proxy (via attribute rewriting)
4. Connection refused returns 502 with a meaningful message
5. Query strings are preserved through the proxy
6. Upstream redirects stay within the proxy path
7. Inline script `fetch()` calls are a known limitation (documented, not fixed in v1)
8. SSRF prevention still blocks dangerous ports
9. All tests pass, `pnpm build` passes
