# Self-Update Feature -- Implementation Plan

## High-Level Goal

Allow the user to update the deployed PhoneCC app from the settings page without SSHing into the VPS. A standalone updater service runs alongside the Next.js app and handles `git pull`, `pnpm install`, `pnpm build`, and systemd service restarts. The frontend shows progress and automatically reconnects when the server comes back.

## Important Technical Notes

### The restart problem
The Next.js server dies when `systemctl restart phonecc` runs. The frontend must handle this:
1. After triggering the update, the frontend switches to a polling loop
2. It tries `GET /api/update` (or any lightweight endpoint) every 2 seconds
3. Once the server responds, it reloads the page
4. If the server does not come back within 120 seconds, show an error: "Server did not come back. SSH in to investigate."

### Concurrency guard
The updater must reject concurrent update requests. A simple in-memory `isUpdating` boolean flag is sufficient (single process). If an update is already in progress, `POST /update` returns `409 Conflict`.

### Pipeline abort on failure
If any step in the update pipeline fails (non-zero exit), the pipeline MUST stop immediately. The streaming response sends an `error` step and ends. The systemd restart NEVER fires if the build failed. This prevents serving a broken build.

### git pull strategy
The VPS clone is treated as read-only. Use `git reset --hard origin/<branch>` after `git fetch` instead of `git pull` to avoid merge conflicts from accidental local edits.

### Branch detection
Use `git rev-parse --abbrev-ref HEAD` to detect the current branch rather than hardcoding `main`. Compare `HEAD` vs `origin/<branch>` using `git rev-list --count HEAD..origin/<branch>`.

### Updater protocol
The updater is a plain HTTP server. Two endpoints:

- `GET /status` -- Runs `git fetch` (with 10s timeout via `AbortSignal.timeout` on the child process), then compares local vs remote HEAD. Returns `RemoteStatus` JSON. Blocked during an active update (returns `409`).
- `POST /update` -- Runs the full update pipeline. Returns a streaming response (line-delimited JSON) so the frontend can show progress. Steps: `git fetch` -> `git reset --hard` -> `pnpm install` -> `pnpm build` -> `systemctl restart`.

### Sudoers
The `phonecc` user needs passwordless sudo for exactly two commands. Added via `/etc/sudoers.d/phonecc-updater`:
```
phonecc ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart phonecc, /usr/bin/systemctl restart phonecc-ws
```

### Restart ordering
Restart `phonecc` (Next.js) first, then `phonecc-ws` (Deepgram WS). The Deepgram WS server runs from source via `tsx`, not from the `.next` build, so it could be skipped on most updates. However, for simplicity, restart both every time since the user is already waiting.

### Local development
The updater is production-only. Do NOT add it to `dev:all` -- running `git reset --hard` on a dev clone would be destructive. Add only a standalone `dev:updater` script for manual testing.

### Security
- Updater binds to `127.0.0.1:9473` only -- not reachable from the internet
- The Next.js API route `/api/update` is protected by the existing auth proxy (`src/proxy.ts`)
- UFW does not open port 9473

## Per-File Methods and Types

### updater.ts (NEW)

Standalone HTTP server that performs git/build/restart operations. Listens on `127.0.0.1:9473`.

- Finds a free port starting from `UPDATER_PORT` env var (default 9473)
- Writes port to `.updater-port` file for discovery (same pattern as deepgram-ws.ts)
- Maintains an `isUpdating` flag to prevent concurrent updates

#### Types

```
UpdateStep {
  step: string          // "git_fetch" | "git_reset" | "pnpm_install" | "pnpm_build" | "restart"
  status: string        // "running" | "done" | "error"
  message: string       // Human-readable status line
}

RemoteStatus {
  upToDate: boolean
  currentCommit: string
  remoteCommit: string
  commitsBehind: number
}
```

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `main` | none | `void` | Starts the HTTP server, sets up routes, writes port file, registers cleanup handlers |
| `handleStatus` | `req: IncomingMessage, res: ServerResponse` | `void` | Runs `git fetch` (10s timeout via `runCommand`) + `git rev-parse` to compare local vs remote HEAD. Returns `RemoteStatus` as JSON. Returns 409 if update is in progress |
| `handleUpdate` | `req: IncomingMessage, res: ServerResponse` | `void` | Checks `isUpdating` flag (409 if busy). Sets flag in try/finally block, runs the full pipeline streaming `UpdateStep` objects as NDJSON (`application/x-ndjson`). Clears flag on completion or error. Aborts pipeline on first failure |
| `runCommand` | `cmd: string, args: string[], timeoutMs?: number` | `Promise<{ stdout: string, exitCode: number }>` | Spawns a child process, captures output. Throws on non-zero exit. Optional timeout via `AbortSignal.timeout` |
| `detectBranch` | none | `Promise<string>` | Returns current branch name via `git rev-parse --abbrev-ref HEAD` |

### api/update/route.ts (NEW)

Next.js API route that proxies requests to the local updater service.

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `GET` | `NextRequest` | `NextResponse` | Reads `.updater-port`, fetches `http://127.0.0.1:{port}/status`, returns the JSON response. Returns 503 if updater is unreachable |
| `POST` | `NextRequest` | `NextResponse` | Reads `.updater-port`, fetches `http://127.0.0.1:{port}/update` (POST), streams the NDJSON response back to the client via `ReadableStream` with `Content-Type: application/x-ndjson` |

### settings/page.tsx (MODIFIED)

Add a "System" section at the bottom of the settings page.

#### New State

```
updateStatus: "idle" | "checking" | "up-to-date" | "update-available" | "updating" | "restarting" | "error"
updateInfo: RemoteStatus | null
updateLog: UpdateStep[]
updateError: string | null
```

#### New Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `handleCheckUpdate` | none | `void` | Calls `GET /api/update`, sets status to "up-to-date" or "update-available". Handles 409 (update in progress) by showing "updating" state |
| `handleStartUpdate` | none | `void` | Calls `POST /api/update`, reads the streamed response, populates `updateLog`. When stream ends or connection drops, switches to "restarting" and starts `pollUntilBack` |
| `pollUntilBack` | none | `void` | Polls `GET /api/update` every 2 seconds. On success, reloads the page. After 120 seconds, gives up and shows error message |

## Phases

### Phase 1: Updater service

1. Create `src/server/updater.ts`:
   - HTTP server on `127.0.0.1:9473` with port discovery
   - `isUpdating` concurrency guard
   - `GET /status` endpoint (git fetch with 10s timeout + compare commits)
   - `POST /update` endpoint (streaming pipeline: fetch, reset --hard, install, build, restart)
   - `runCommand` helper using `child_process.spawn`
   - `detectBranch` helper
   - Port file at `.updater-port`
   - Signal handlers for cleanup
2. Add `"dev:updater": "tsx src/server/updater.ts"` script to `package.json`
3. Add `.updater-port` to `.gitignore`

### Phase 2: API route

1. Create `src/app/api/update/route.ts`:
   - `GET` handler: read `.updater-port`, proxy to updater `/status`, return JSON
   - `POST` handler: read `.updater-port`, proxy to updater `/update`, stream response back via `ReadableStream`

### Phase 3: Settings UI

1. Add "System" section at the bottom of `settings/page.tsx`
2. "Check for updates" button that calls `GET /api/update`
3. Display update info (commits behind, current/remote commit short hashes)
4. "Update now" button that calls `POST /api/update`
5. Progress display: show each `UpdateStep` as it streams in
6. "Restarting..." state with polling loop (2s interval, 120s timeout)
7. Auto-reload when server comes back, or error message if timeout

### Phase 4: Deployment setup

1. Update `INSTALL.md` with new phase:
   - Sudoers file at `/etc/sudoers.d/phonecc-updater`
   - New systemd service `phonecc-updater.service`
   - Enable the service
2. Update `INSTALL.md` to note the VPS clone is read-only (no local edits)

## Success Criteria

- User can check for updates from the settings page without SSH access
- User can trigger a full update (pull, install, build, restart) from the settings page
- Frontend shows progress during the update and reconnects after restart
- Concurrent update requests are rejected
- Pipeline aborts on failure -- broken builds never get restarted
- Polling gives up after 120 seconds with a clear error
- Updater service is only reachable from localhost
- Updater service survives the Next.js restart
- Existing auth protects the update endpoint
