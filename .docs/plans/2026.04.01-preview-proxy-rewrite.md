# Preview Proxy via NextResponse.rewrite()

## High-Level Goal

Replace the broken sub-path proxy (`/preview/{token}/...`) with a `NextResponse.rewrite()` approach in `proxy.ts`. When a user clicks a preview link, a cookie is set identifying the active preview port. While that cookie is set, proxy.ts rewrites ALL requests to `localhost:{port}`, making the upstream app think it owns the entire origin. This solves the absolute-path problem because `/_next/static/...` requests also go through proxy.ts and get rewritten to the upstream.

The flow:
1. Agent says "dev server at http://localhost:3002" -- rewritten to `/preview/{token}` in chat (unchanged)
2. User clicks the preview link -- hits `/preview/{token}` route, which validates the token, sets a `phonecc_preview={port}` cookie, and redirects to `/`
3. proxy.ts sees the preview cookie and rewrites ALL requests to `localhost:{port}`
4. User sees the upstream app rendered correctly, including all assets and API calls
5. User navigates back to PhoneCC (via a "back to PhoneCC" button or clearing the cookie) -- cookie is removed, normal PhoneCC operation resumes

Limitation: only one preview can be active per browser at a time (because the cookie applies to the whole origin).

## Important Technical Notes

### proxy.ts matcher must NOT exclude `_next/static`

The current proxy.ts matcher excludes `_next/static` and `_next/image`:
```
"/((?!_next/static|_next/image|favicon.ico|...)*)"
```

For the preview rewrite to work, proxy.ts must intercept `_next/static` requests too (when the preview cookie is set). Use conditional matchers with the `has` condition:

```js
matcher: [
  // When preview cookie is set, match everything (including _next/static)
  { source: "/(.*)", has: [{ type: "cookie", key: "phonecc_preview" }] },
  // Normal auth matcher (existing, unchanged)
  "/((?!_next/static|_next/image|favicon.ico|icon-192.png|manifest.json|login(?:/|$)|api/auth).*)",
]
```

This avoids running proxy.ts on every static asset request when no preview is active, while still intercepting them during preview mode. Next.js 16 supports `has`/`missing` conditions in matchers (see `proxy.md` docs).

### Preview cookie scope

The `phonecc_preview` cookie stores the port number (e.g., `3002`). It uses `path: "/"` so it applies to all requests. The cookie is HttpOnly (upstream JS cannot read or tamper with it). The cookie has `maxAge: 14400` (4 hours) to auto-expire stale previews if the user forgets to exit.

To signal preview mode to the frontend (for the exit banner), proxy.ts injects an `x-phonecc-preview` response header when rewriting. The chat-view component does NOT need to detect preview mode -- see "Exit mechanism" below for why.

### SSRF prevention: defense in depth

Port validation happens in TWO places:
1. **Route handler** (`/preview/{token}`): validates token and checks port before setting the cookie
2. **proxy.ts** (`handlePreviewRewrite`): validates the port from the cookie before rewriting

The route handler is the primary gate, but proxy.ts ALSO validates because cookies are user-controlled -- a malicious user could manually set `phonecc_preview=22` via DevTools. proxy.ts must reject:
- Non-numeric cookie values (NaN check after `parseInt`)
- Ports < 1024 (privileged)
- Port 3000 / APP_PORT (request loop)
- Ports > 65535 (invalid)

### Exit mechanism

The user needs a way to stop previewing and return to PhoneCC. A request to `/preview/exit` clears the `phonecc_preview` cookie and redirects to `/`.

**Critical: proxy.ts must exempt `/preview/*` paths from the rewrite.** If `/preview/exit` gets rewritten to the upstream, the user is trapped with no way to clear the cookie. proxy.ts checks the pathname BEFORE rewriting: if it starts with `/preview/`, skip the rewrite and let Next.js route it normally.

**Exit banner injection**: Since the PhoneCC UI does NOT render during preview (all requests go to the upstream), the "Back to PhoneCC" banner must be injected by proxy.ts into the rewritten HTML response. When `handlePreviewRewrite` processes an HTML response, it injects a small fixed-position banner (`<div>` + inline styles + link to `/preview/exit`) at the end of `<body>`. This is fragile but acceptable for v1. Non-HTML responses are not modified.

### Route priority: `/preview/exit` vs `/preview/[token]`

Next.js resolves static routes before dynamic segments, so `/preview/exit/route.ts` will be matched for `/preview/exit` before `/preview/[token]/[[...path]]/route.ts`. This is standard Next.js behavior. If for any reason this doesn't work in Next.js 16, fallback: handle `token === "exit"` inside the existing `[token]` route handler before the `lookupToken` call.

### WebSocket limitation

`NextResponse.rewrite()` does NOT proxy WebSocket connections. This means Next.js HMR (Hot Module Replacement) will not work in the previewed app. The user must manually refresh the page to see changes. This is an accepted v1 limitation.

### Multiple previews

Only one preview can be active at a time per browser. If the user clicks a second preview link, the cookie is overwritten with the new port. This is acceptable for v1 since the phone interface only has one viewport.

### RSC header propagation

When using `NextResponse.rewrite()` to proxy to another Next.js app, Next.js 16 automatically propagates RSC (React Server Components) headers. This is important because the upstream app may also be a Next.js app.

## Per-File Methods and Types

### proxy.ts (MOD)

File description: Next.js 16 proxy for authentication and preview rewriting. Intercepts all requests. When a preview cookie is active, rewrites requests to the upstream dev server. Otherwise, performs token-based authentication.

#### Constants

```
PREVIEW_COOKIE_NAME = "phonecc_preview"
```

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `proxy` | `request: NextRequest` | `Promise<NextResponse>` | Main proxy function. If path starts with `/preview/`, skip rewrite (let route handler handle it). Check for preview cookie -- if present AND port is safe, rewrite to `localhost:{port}`. Otherwise runs existing auth logic |
| `handlePreviewRewrite` | `request: NextRequest, port: number` | `NextResponse \| null` | Validates port (>= 1024, <= 65535, != APP_PORT, not NaN). Builds the rewrite URL (`http://localhost:{port}` + original pathname + search), returns `NextResponse.rewrite(url)`. Returns `null` if port is unsafe (caller falls through to normal auth) |
| `isPortSafe` | `port: number` | `boolean` | Returns true if port is a valid, non-privileged, non-APP_PORT number. Used by both proxy.ts and route.ts |

### route.ts (preview/[token]/[[...path]]) (MOD)

File description: Preview entry point. Validates the token, sets the preview cookie, and redirects to `/`. Also handles the exit flow to clear the cookie.

Simplify drastically -- remove all the fetch-based proxy logic, HTML rewriting, Location header rewriting. This file becomes ~40 lines.

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `GET` | `request: Request, ctx: RouteContext` | `Response` | Validates token, checks port safety (>= 1024, != APP_PORT), sets `phonecc_preview={port}` cookie with `maxAge: 14400`, redirects to `/` |

### exit/route.ts (preview/exit) (NEW)

File description: Clears the preview cookie and redirects to `/`. Separate route so it doesn't depend on a token parameter.

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `GET` | `request: Request` | `Response` | Clears the `phonecc_preview` cookie (set maxAge to 0), redirects to `/` |

### preview-manager.ts (MOD)

Changes:
- Remove debug `console.log` statements
- Export `PREVIEW_COOKIE_NAME` constant for use by proxy.ts and route.ts

### chat-view.tsx (UNCHANGED)

The `resolvePreviewUrls` function already generates `/preview/{token}` URLs. These will now function as entry points that set the cookie and redirect. No change needed. The exit banner is injected by proxy.ts, not rendered by chat-view.

## Phases

### Phase 1: Update proxy.ts matcher and add preview rewrite ✅
- Add conditional matcher: `{ source: "/(.*)", has: [{ type: "cookie", key: "phonecc_preview" }] }` to match ALL requests when preview cookie is set (existing auth matcher stays as-is)
- Exempt `/preview/*` paths from rewrite (let Next.js route handler handle them)
- Add `isPortSafe(port)`: validates port is numeric, >= 1024, <= 65535, != APP_PORT
- Add `handlePreviewRewrite`: read `phonecc_preview` cookie, validate port via `isPortSafe`, rewrite to `http://localhost:{port}` + original path + query string. Return `null` if unsafe (fall through to auth)
- Preview rewrite runs BEFORE auth check. Auth is intentionally skipped for preview requests because: (a) the preview cookie can only be set via the auth-gated PhoneCC UI, (b) the cookie is HttpOnly so upstream apps can't read it, (c) the port validation in proxy.ts prevents SSRF even with forged cookies. The worst case for a forged cookie is accessing a dev server on a high port, which is acceptable since PhoneCC runs on a single-user VPS

## Success Criteria

1. Clicking a preview URL renders the upstream app correctly, including all JS/CSS/image assets
2. Upstream errors (like Supabase missing-env) display properly with the Next.js error overlay
3. API calls from the upstream app work correctly (same origin)
4. User can return to PhoneCC by clicking the exit banner
5. SSRF prevention blocks dangerous ports
6. Auth still works correctly for non-preview requests
7. `_next/static` requests are not intercepted when no preview cookie is set (conditional matcher)
8. All tests pass, `pnpm build` passes
