# Token-Based Authentication

## High-Level Goal

Add a zero-config authentication system so the app is secure when deployed to a VPS. On first startup, a random access token is generated and printed to the terminal. Users visit the app with `?token=<token>` in the URL, which sets an HttpOnly cookie. All subsequent requests are authenticated via that cookie. No external services, no OAuth apps, no user setup beyond running the app.

## Tests

No testing framework is set up in this project. The plan does not include test infrastructure setup — that is a separate concern. If tests are desired, vitest should be added first.

Manual verification:
- Start the app fresh (no `~/.phonecc/auth-token` file) → token is printed to terminal
- Visit any page without cookie → redirected to `/login`
- Visit `/login?token=<correct-token>` → cookie is set, redirected to `/`
- Visit any page with valid cookie → access granted
- Visit any API route without cookie → 401 JSON response
- Delete `~/.phonecc/auth-token` and restart → new token generated, old cookies invalidated
- SSE endpoint (`/api/sessions/[id]/stream`) works with valid cookie

## Important Technical Notes

### Next.js 16: proxy.ts, not middleware.ts

Next.js 16 renamed middleware to "proxy". The file must be `src/proxy.ts`, the export must be named `proxy`, and it runs on the Node.js runtime only (not edge). Cookies are accessed via `request.cookies.get()` on `NextRequest`, not the async `cookies()` from `next/headers`.

### Token auto-detection on URL

When a user visits any page with `?token=<token>` in the URL, the proxy should validate it and set the cookie immediately, then redirect to the same URL without the token query param. This way the token is never visible in browser history after the first load.

### Cookie security

The auth cookie must be:
- `HttpOnly` — not accessible via JavaScript
- `Secure` — only sent over HTTPS (skip in dev with `NODE_ENV=development`)
- `SameSite=Lax` — prevents CSRF on POST but allows top-level navigations (needed for `?token=` URL flow from external links like email/Slack)
- Long-lived (e.g. 1 year) — users shouldn't need to re-authenticate frequently

### Token generation on startup

The token should be generated via a Next.js instrumentation hook (`instrumentation.ts`) or checked lazily on first request in `auth.ts`. Since Next.js `pnpm start` doesn't run custom startup scripts, the lazy approach is simpler — on first call to `getOrCreateToken()`, check if `~/.phonecc/auth-token` exists. If not, generate one, save it, and log it to stdout.

The token must be cached in a module-level variable after the first disk read to avoid filesystem I/O on every proxy invocation. Use `ensurePhoneccDir()` (same pattern as `projects.ts`) before writing the token file.

**Deployment constraint:** This approach (shared module-level state between proxy and route handlers) only works on self-hosted Node.js deployments. It will NOT work on CDN/edge deployments where the proxy runs in a separate process.

### Cookie value

The cookie value is the raw token itself. The proxy validates it against the stored (in-memory cached) token on every request using `validateToken()` with timing-safe comparison. The 256-bit token space (2^256 possible values) makes brute-force infeasible even without rate limiting on the proxy's `?token=` URL flow — rate limiting is only applied to the `/api/auth/validate` endpoint where manual entry occurs.

### Rate limiting on token validation

The `/api/auth/validate` endpoint must include a simple in-memory rate limiter (e.g., max 5 attempts per IP per minute) to prevent brute-force attacks. Failed validation attempts should be logged with `console.warn`.

### Deepgram WebSocket server

The standalone WebSocket server at `src/server/deepgram-ws.ts` (port 3001) proxies audio to Deepgram's API to keep the API key server-side. It must be bound to `127.0.0.1` (localhost only) so it is not reachable from the internet — only the Next.js app on the same machine can connect to it.

## Per-File Methods and Types

### auth.ts (NEW)

#### Types

```
AuthConfig {
  tokenPath: string        // ~/.phonecc/auth-token
  cookieName: string       // "phonecc_auth"
  cookieMaxAge: number     // seconds (default: 365 days)
}
```

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `getOrCreateToken` | none | `Promise<string>` | Reads token from `~/.phonecc/auth-token`. If file doesn't exist, generates a 256-bit random hex token prefixed with `phcc_`, saves it, logs it to stdout, and returns it |
| `validateToken` | `token: string` | `Promise<boolean>` | Compares provided token against stored token using timing-safe comparison |
| `COOKIE_NAME` | — | `string` (constant) | The cookie name: `"phonecc_auth"` |
| `COOKIE_MAX_AGE` | — | `number` (constant) | Cookie max age in seconds |

### proxy.ts (NEW)

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `proxy` | `request: NextRequest` | `Promise<NextResponse>` | Async. Checks for auth cookie or `?token=` query param. If token in URL: validate, set cookie, redirect without token. If valid cookie: pass through. Otherwise: redirect to `/login` (pages) or return 401 (API routes) |

#### Config

```ts
export const config = {
  matcher: [
    // Match everything except static files and the login page itself
    '/((?!_next/static|_next/image|favicon.ico|icon-192.png|manifest.json|login(?:/|$)|api/auth).*)',
  ],
}
```

### login/page.tsx (NEW)

Client component with:
- A text input for pasting the token
- A submit button that POSTs to `/api/auth/validate`
- On success, redirects to `/`
- Shows error message on invalid token
- Minimal styling consistent with existing app theme

### api/auth/validate/route.ts (NEW)

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `POST` | `Request` (body: `{ token: string }`) | `Response` | Validates the token with in-memory rate limiting (5 attempts/IP/min). If valid, returns 200 with `Set-Cookie` header. If invalid, returns 401. Logs failed attempts with `console.warn` |

## Phases

- [ ] Create `src/lib/auth.ts` with `getOrCreateToken()`, `validateToken()`, and constants
- [ ] Uses `~/.phonecc/auth-token` for persistence, `crypto.randomBytes(32)` for generation

- [ ] Create `src/proxy.ts` with the `proxy` function
- [ ] Check for `phonecc_auth` cookie → if valid, `NextResponse.next()`

- [ ] For API requests without auth → return 401 JSON
- [ ] Matcher excludes `_next/static`, `_next/image`, static assets, `/login`, `/api/auth`, and `manifest.json`

- [ ] In `src/server/deepgram-ws.ts`, change `httpServer.listen(PORT)` to `httpServer.listen(PORT, "127.0.0.1")` so it only accepts local connections

### Phase 5: Verify

## Success Criteria

1. App generates and prints access token on first run with zero user configuration
2. All pages and API routes are protected — unauthenticated requests get redirected (pages) or 401 (APIs)
3. Token can be provided via URL query param or login page — both set an HttpOnly cookie
4. Subsequent visits with a valid cookie require no further authentication
5. Deleting `~/.phonecc/auth-token` and restarting invalidates all sessions
6. No external dependencies added (uses only `crypto` and `fs` from Node.js)
7. Build passes
