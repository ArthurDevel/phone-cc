# Feature 3: Session Backend — Implementation Plan

## Overview

Build server-side session management: creating sessions (clone repo, spawn Claude agent), listing sessions, closing sessions (with unpushed commit safety), and reconnecting to sessions. Uses `@anthropic-ai/claude-agent-sdk` for the agent process.

**Important SDK note:** The programmatic SDK is `@anthropic-ai/claude-agent-sdk` (not `@anthropic-ai/claude-code` which is just the CLI). The PROMPT.md references the old name.

## Step 4: Session Manager (Core Logic)

**File:** `src/lib/session-manager.ts`

Module-level singleton that manages:
- In-memory map: `Map<string, { query: Query; abortController: AbortController; sdkSessionId: string }>`
- All session CRUD operations

### Constants
- `SESSIONS_DIR = path.join(os.homedir(), '.phonecc', 'sessions')`
- `MAX_SESSIONS = 5`

### Functions

#### `generateBranchName(repoUrl: string): Promise<string>`
1. Pick random city from CITIES array
2. Check local session directories for existing names
3. Run `git ls-remote --heads {repoUrl}` to check remote branches
4. If conflict, append `-2`, `-3`, etc. until unique
5. Return the unique branch name

#### `createSession(projectId: string): Promise<Session>`
1. Read project from `projects.json` via `readProjects()`. 404 if not found.
2. Count existing session directories. 409 if >= 5.
3. Generate unique branch name via `generateBranchName()`.
4. Build clone URL with GitHub PAT: parse `repoUrl` to extract host/path, embed PAT from `process.env.GITHUB_PERSONAL_ACCESS_TOKEN`.
5. `git clone --branch {defaultBranch} {cloneUrl} {sessionDir}`
6. `git checkout -b {branchName}` in the cloned dir
7. Write `session.json` with metadata
8. Spawn Claude agent via `spawnAgent(branchName, sessionDir)`
9. Return session with `status: "active"`

#### `spawnAgent(sessionId: string, cwd: string, resumeSessionId?: string): void`
1. Create `AbortController`
2. Call `query()` from `@anthropic-ai/claude-agent-sdk`:
   ```ts
   const q = query({
     prompt: createAsyncIterable(), // streaming input mode
     options: {
       cwd,
       abortController,
       permissionMode: "bypassPermissions",
       allowedTools: ["Read", "Edit", "Write", "Bash", "Glob", "Grep"],
       resume: resumeSessionId, // undefined for new sessions
       includePartialMessages: true,
     }
   });
   ```
3. Store `{ query: q, abortController, sdkSessionId: "" }` in the map
4. Start consuming the async generator in background, capturing `sdkSessionId` from the init system message
5. The async iterable input allows us to send messages later via `streamInput()`

**Note on streaming input mode:** The `query()` function accepts `prompt: AsyncIterable<SDKUserMessage>` for multi-turn conversations. We create an async iterable that we can push messages into. Alternatively, for each user message, we can call a new `query()` with `resume: sdkSessionId`.

**Simpler approach:** For each user message, call a new `query()` with `resume: sdkSessionId`. This avoids managing a persistent async iterable. The session ID from the SDK ensures continuity.

#### `listSessions(): Promise<Session[]>`
1. Read directories in `SESSIONS_DIR`
2. For each, read `session.json`. Skip on error.
3. Check in-memory map for status: `active` if present, `disconnected` if not
4. Sort by `createdAt` descending
5. Return array

#### `closeSession(id: string, confirmed: boolean): Promise<{ deleted: boolean; warning?: object }>`
1. Check session dir exists. Return null if not.
2. If not confirmed:
   - Run `git log origin/{branchName}..HEAD --oneline` in session dir
   - If output is non-empty, return `{ deleted: false, requiresConfirmation: true, warning: "unpushed_commits", count, commits }`
3. If confirmed or no unpushed commits:
   - Kill agent: call `abortController.abort()` and remove from map
   - Delete session directory recursively (`fs.rm` with `recursive: true, force: true`)
   - Return `{ deleted: true }`

#### `reconnectSession(id: string): Promise<Session | null>`
1. Check session dir exists. Return null if not.
2. If already in the in-memory map, return session with `status: "active"`
3. Read `session.json` for metadata
4. Read the SDK session ID from the last session state (stored in session dir as `.sdk-session-id`)
5. Spawn agent with `resume: sdkSessionId`
6. Return session with `status: "active"`

#### `getAgent(sessionId: string)` — returns the agent entry from the map or null

#### Helper: persist SDK session ID
When spawning an agent, after receiving the init system message with `session_id`, write it to `{sessionDir}/.sdk-session-id` for later resume.

## Step 5: API Route — GET/POST `/api/sessions`

**File:** `src/app/api/sessions/route.ts`

### GET handler
- Call `listSessions()`
- Return `Response.json({ sessions })`

### POST handler
- Parse body: `{ projectId }`
- Validate: projectId must be non-empty string. Return 400 if invalid.
- Call `createSession(projectId)`
- Handle errors:
  - Project not found → 404
  - Max sessions → 409
  - Clone failure → 500
- Return `Response.json(session)` with status 201

---

## Step 6: API Route — DELETE `/api/sessions/[id]`

**File:** `src/app/api/sessions/[id]/route.ts`

### DELETE handler
- Extract `id` from `await ctx.params`
- Parse body for `{ confirmed }` (may be empty body for first call)
- Call `closeSession(id, confirmed)`
- If session not found → 404
- If needs confirmation → 200 with `{ requiresConfirmation: true, warning, count, commits }`
- If deleted → 200 with `{ success: true }`

---

## Step 7: API Route — POST `/api/sessions/[id]/reconnect`

**File:** `src/app/api/sessions/[id]/reconnect/route.ts`

### POST handler
- Extract `id` from `await ctx.params`
- Call `reconnectSession(id)`
- If not found → 404
- Return `Response.json(session)`

## Step 9: API Verification (curl)

**Prerequisites:** Need at least one project in `projects.json` and a valid GitHub PAT in `.env.local`.

```bash
# Create a project first
curl -X POST http://localhost:3000/api/projects \
  -H "Content-Type: application/json" \
  -d '{"name":"Test","repoUrl":"https://github.com/user/repo","defaultBranch":"main"}'

# POST — create session (need the projectId from above)
curl -X POST http://localhost:3000/api/sessions \
  -H "Content-Type: application/json" \
  -d '{"projectId":"<uuid>"}'
# Expect: 201, session metadata with status "active"

# GET — list sessions
curl http://localhost:3000/api/sessions
# Expect: 200, { sessions: [...] }

# DELETE — close session (first call, check unpushed)
curl -X DELETE http://localhost:3000/api/sessions/<id>
# Expect: 200, either { success: true } or { requiresConfirmation: true, ... }

# DELETE — confirmed close
curl -X DELETE http://localhost:3000/api/sessions/<id> \
  -H "Content-Type: application/json" \
  -d '{"confirmed":true}'
# Expect: 200, { success: true }

# POST — reconnect
curl -X POST http://localhost:3000/api/sessions/<id>/reconnect
# Expect: 200 with status "active", or 404
```

**Note:** Full curl verification requires a valid GitHub repo URL and PAT. For build verification, the code must compile without errors even without runtime credentials.

## Key Design Decisions

1. **One `query()` per user message** rather than a persistent streaming connection. Each message creates a new `query()` call with `resume: sdkSessionId`, letting the SDK handle conversation continuity. This is simpler and more resilient than managing a long-lived async iterable.

2. **SDK session ID stored on disk** at `{sessionDir}/.sdk-session-id`. This survives server restarts and enables resume.

3. **`permissionMode: "bypassPermissions"`** since this is a local single-user app — the user trusts the agent to perform actions.

4. **Agent lifecycle tied to message processing**, not a persistent background process. The agent runs while processing a message and the in-memory map tracks whether a session has an active agent.
