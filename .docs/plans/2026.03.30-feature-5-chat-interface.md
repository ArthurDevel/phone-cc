# Feature 5: Chat Interface — Implementation Plan

## Overview

Build the chat UI with message bubbles (user/assistant), tool use collapsed rows with detail modal, SSE streaming from the backend, message sending, history loading, and auto-scroll. This requires backend API routes (message, stream, history) and frontend components.

## Step 2: Session Manager Updates

**Modify:** `src/lib/session-manager.ts`

### Changes needed

The current `spawnAgent` approach spawns once with a single prompt and `maxTurns: 1`. For chat, we need a different pattern:

**Per-message query approach:** When a user sends a message, we call `query()` with that message as the prompt, resuming the SDK session. The async generator yields streaming events. We need to:

1. Add an **SSE event emitter** per session so multiple SSE clients can listen.
2. Add a `sendMessage(sessionId, text)` function that:
   - Creates a new `query()` call with `resume: sdkSessionId` and the user's text as `prompt`
   - Iterates the async generator, emitting SSE events for each SDK message
   - Updates the agent entry in the map

### New exports
- `sendMessage(sessionId: string, text: string): Promise<void>` — starts a new query with the message
- `getEventEmitter(sessionId: string)` — returns an EventEmitter for SSE streaming
- `getMessageHistory(sessionId: string): Promise<Message[]>` — reads SDK session JSONL

### EventEmitter approach
Use a simple custom event system (or Node's EventEmitter) per session. The SSE route subscribes, the message processing loop emits events.

Store per session:
```ts
interface AgentEntry {
  sdkSessionId: string;
  abortController: AbortController;
  query: Query | null;
  emitter: EventEmitter;
  processing: boolean;
}
```

### Message processing flow
When `sendMessage()` is called:
1. Set `processing = true`
2. Emit `status_change` event with `{ status: "thinking" }`
3. Create new `query({ prompt: text, options: { resume: sdkSessionId, cwd, ... } })`
4. Iterate the generator:
   - `type === "system"` (init): capture `session_id`, save to disk
   - `type === "stream_event"`: parse the `BetaRawMessageStreamEvent`:
     - `content_block_start` with `type: "text"` → nothing special
     - `content_block_delta` with `type: "text_delta"` → emit `text_delta` event
     - `content_block_start` with `type: "tool_use"` → emit `tool_use_start` with name and input
     - `content_block_stop` → nothing
   - `type === "assistant"`: complete message — extract tool results from the following user message
   - `type === "user"` with `tool_use_result`: emit `tool_use_result` events
   - `type === "result"`: emit `message_end`, set `processing = false`, emit `status_change` with `{ status: "idle" }`
5. On error: emit `status_change` with `{ status: "error" }`

### History reading
Read the SDK's JSONL session file at `~/.claude/projects/<encoded-cwd>/<session-id>.jsonl`. Parse each line, extract user and assistant messages, convert tool_use content blocks to our ToolUse format.

Actually, a simpler approach: maintain our own message history in memory per session, built up from the streaming events. For history on reconnect, read the SDK's session state.

**Simplest approach for history:** Keep a `messages: Message[]` array in each `AgentEntry`. Append to it during streaming. On reconnect, reconstruct from SDK session file or start fresh.

## Step 3: SSE Endpoint

**File:** `src/app/api/sessions/[id]/stream/route.ts`

### GET handler
- Extract session `id`
- Get the event emitter for this session from the session manager
- Return a `ReadableStream` response with `Content-Type: text/event-stream`
- Subscribe to emitter events, write SSE formatted data (`data: {...}\n\n`)
- Keep connection open until client disconnects (AbortSignal)
- Event format: `event: <type>\ndata: <json>\n\n`

---

## Step 4: Message Send Endpoint

**File:** `src/app/api/sessions/[id]/message/route.ts`

### POST handler
- Extract session `id` and body `{ text }`
- Validate text is non-empty string. 400 if not.
- Check session exists. 404 if not.
- Call `sendMessage(id, text)` (fire-and-forget — response streams via SSE)
- Return 200 immediately with `{ ok: true }`

---

## Step 5: History Endpoint

**File:** `src/app/api/sessions/[id]/history/route.ts`

### GET handler
- Extract session `id`
- Call `getMessageHistory(id)` from session manager
- Return `{ messages: Message[] }`
- 404 if session not found

## Step 6: Chat View Component

**File:** `src/components/chat-view.tsx`

`"use client"` component that renders the chat for the active session.

### Props/Context
- Uses `useSession()` to get `activeSessionId`

### State
- `messages: Message[]` — current chat messages
- `showScrollButton: boolean` — user scrolled up
- `toolModal: ToolUse | null` — which tool use detail to show

### Behavior
- **On session switch (`activeSessionId` changes):**
  1. Fetch `GET /api/sessions/{id}/history` to populate messages
  2. Connect to SSE `GET /api/sessions/{id}/stream`
  3. Process SSE events to update messages in real-time
- **On unmount / session switch:** close previous EventSource
- **SSE event handling:**
  - `text_delta`: append text to last assistant message's content (or create new assistant message)
  - `tool_use_start`: add new ToolUse to last assistant message
  - `tool_use_result`: update the ToolUse with output and status
  - `message_end`: finalize current message
  - `status_change`: update session status in context

### Auto-scroll
- `scrollContainerRef` on the main chat div
- After each message update, if `isAtBottom` (scrollTop + clientHeight >= scrollHeight - 100), scroll to bottom
- On scroll: check if user is at bottom. If not, show scroll-to-bottom button.
- Scroll button: fixed position pill, "↓" icon, on click scroll to bottom

## Step 9: Simple Markdown Renderer

**File:** `src/lib/markdown.ts`

A simple function `renderMarkdown(text: string): string` that converts markdown to HTML using regex:
- Bold: `**text**` → `<strong>text</strong>`
- Italic: `*text*` → `<em>text</em>`
- Inline code: `` `code` `` → `<code>code</code>`
- Code blocks: ```` ```lang\ncode\n``` ```` → `<pre><code>code</code></pre>`
- Bullet lists: lines starting with `- ` or `* `
- Numbered lists: lines starting with `\d+. `
- Line breaks: `\n` → `<br>`

Return sanitized HTML string. Use `dangerouslySetInnerHTML` in the bubble component (safe since content comes from Claude, not user input).
