# Update Available Banner -- Implementation Plan

## High-Level Goal

Show a non-intrusive banner on the home page when a newer version of PhoneCC is available. The app checks for updates on page load but caches the result for 1 hour to avoid hammering the updater service. Clicking the banner navigates the user to the settings page where they can apply the update. The banner can be dismissed per-update (keyed by remote commit SHA).

## Important Technical Notes

### Cache strategy

Use `localStorage` with two keys:
- `phonecc:update-check` -- JSON with `{ checkedAt: number, status: RemoteStatus }`. The hook reads this on mount. If `Date.now() - checkedAt < CACHE_TTL_MS` (1 hour), it uses the cached result without fetching.
- `phonecc:update-dismissed` -- The remote commit SHA that was dismissed. When the user clicks the dismiss button, store the current `remoteCommit`. On next check, if the remote commit matches the dismissed one, the hook returns `null`. A new remote commit (new update pushed) will clear the dismissal automatically since the SHA won't match.

### Dismiss returns null, not a flag

When an update is dismissed, `useUpdateCheck` returns `null` (same as up-to-date). This simplifies the consumer -- the banner renders only when the hook returns non-null. The `dismiss` callback is part of the returned result, so the consumer calls `result.dismiss()` before the result becomes `null`. Internally, `dismiss()` updates both localStorage AND a React state variable (`dismissedSha`) to trigger an immediate re-render. The `dismiss` callback should read `remoteCommit` from the hook's internal state (not a stale closure) -- use `useCallback` with the state value as a dependency.

### Fail silently

The banner is a nice-to-have. If the updater service is down (503), the fetch fails, or localStorage is unavailable, the hook should return `null` and the banner simply does not appear. No error toasts or fallback UI.

### Cache scope -- hook only

The cache lives entirely in the browser via `localStorage` and is only used by the `useUpdateCheck` hook on the home page. The settings page's "Check for updates" button calls `GET /api/update` directly via its own `handleCheckUpdate` function -- it does NOT go through the hook, so it always gets a fresh result. No changes to the settings page update logic are needed.

### Banner placement and layout

The update banner renders AFTER the offline banner, before `<ChatView />`. This ensures the offline warning (more urgent, transient) always appears first when both are visible. The banner uses `shrink-0` inside the existing `flex flex-col` container, same as the offline banner. `ChatView` flexes to fill the remaining space, so the page does not gain a scrollbar -- the chat area simply gets slightly shorter.

This file must include `"use client"` at the top since it uses React client APIs (`useState`, `useEffect`) and browser APIs (`localStorage`, `fetch`).

The `src/hooks/` directory does not exist yet -- this is the first extracted hook in the codebase (other hooks like `usePrBadge` live inline in their page files). Extracted here because it needs independent testing.

#### Types

```
UpdateCheckResult {
  commitsBehind: number       // Number of commits the local version is behind
  currentCommit: string       // Current local commit SHA
  remoteCommit: string        // Remote commit SHA

}
```

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `useUpdateCheck` | none | `UpdateCheckResult \| null` | On mount, checks localStorage cache. If stale or missing, fetches `GET /api/update`. Returns `null` if up-to-date, dismissed, or on error. Returns `UpdateCheckResult` if behind and not dismissed. |

## Phases

### Phase 0: Setup test infrastructure ✅

1. Install dev dependencies: `pnpm add -D @testing-library/react jsdom` (do NOT install `@testing-library/react-hooks` -- it is deprecated; `renderHook` ships natively in `@testing-library/react` v13+)
2. Do NOT add `environment: "jsdom"` globally to `vitest.config.ts` -- existing tests (`preview-manager`, etc.) run in Node and would break. Instead, add a `// @vitest-environment jsdom` comment at the top of the hook test file only.

### Phase 1: Extract shared type ✅

1. Create `src/types/update.ts` with the `RemoteStatus` interface
2. Update `src/app/settings/page.tsx` to import `RemoteStatus` from `@/types/update` instead of defining it inline

### Phase 2: Create `useUpdateCheck` hook ✅

1. Create `src/hooks/useUpdateCheck.ts`
2. Define `CACHE_TTL_MS = 3_600_000` (1 hour) and localStorage keys as constants
3. Implement the hook:
   - On mount, read `phonecc:update-check` from localStorage
   - If cache is fresh, use cached `RemoteStatus`
   - If cache is stale or missing, call `GET /api/update`
   - On success, write result + timestamp to localStorage
   - On any error (network, 503, parse), return `null`
   - If `upToDate` is true, return `null`
   - If update available, check `phonecc:update-dismissed` against `remoteCommit`
   - If dismissed, return `null`
   - Return `UpdateCheckResult` with `dismiss` callback
4. The `dismiss` callback writes `remoteCommit` to `phonecc:update-dismissed` AND updates a `dismissedSha` state variable to trigger re-render

### Phase 3: Add banner to home page ✅

1. Import `useUpdateCheck` in `src/app/page.tsx`
2. Call the hook in the `Home` component
3. Add a banner `<div>` after the offline banner, before `<ChatView />`:
   - Only render when result is non-null
   - Style: similar to the offline banner (`px-4 py-2 text-xs text-center shrink-0`) but with `bg-accent/20 text-accent` to distinguish from the warning-colored offline banner
   - Text: "Update available (N commits behind)" as a clickable link to `/settings`
   - Small "X" dismiss button on the right
4. Import `Link` from `next/link` for navigation to `/settings`
5. Define a small inline `DismissIcon` in the icons section of `page.tsx`

### Phase 4: Write tests ✅

1. Create `src/__tests__/useUpdateCheck.test.tsx` with `// @vitest-environment jsdom` at the top
2. Mock `fetch` via `vi.spyOn(global, "fetch")` and clear `localStorage` in `beforeEach`
3. Use `renderHook` from `@testing-library/react`
4. Write the 6 test cases listed in the Tests section

## Success Criteria

- On first page load, the app calls `GET /api/update` and shows the banner if an update is available
- On subsequent page loads within 1 hour, the app does NOT call the API again (uses cache)
- After 1 hour, the app re-checks and updates the cache
- Clicking the banner text navigates to `/settings`
- Clicking the dismiss "X" hides the banner immediately for that specific update
- A new update (different remote commit) shows the banner again even if the previous was dismissed
- If the updater service is down or unreachable, no banner appears and no errors are shown
- The hook returns `null` when the app is up-to-date
- All 6 unit tests pass
